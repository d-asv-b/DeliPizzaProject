### `/api/account/
- `/sign_in`
    `POST` - запрос.

    Параметры:
    -  `email` - электронная почта
    -  `pwdHash` - хэш пароля, созданный с помощью библиотеки `bcrypt`. Длина - 64 символа.

    Ожидаемое поведение:
    -  На сервере сначала выполняется поиск пользователя с такой почтой в БД. После этого из базы данных берется соль, сгенерированная для данного пользователя при регистрации. `pwdHash` хэшируется с помощью алгоритма `argon2` (длина - 128 бит, настройки по умолчанию) и сравнивается с хэшем данного пользователя из БД.
    -  Если пользователь не найден или хэши не совпадают, то возвращается JSON с сообщением "Неправильный логин или пароль" (код ответа - 404 Not Found).
    -  Если пользователь найден, то с помощью встроенного в Django `Sessions framework`-а создается сессия (код ответа - `200 OK`).
- `/sign_up`
    `POST` - запрос.

    Параметры:
    -  `username` - имя пользователя
    -  `email` - эл. почта
    -  `phoneNumber` - номер телефона
    -  `birthdayDate` - дата дня рождения пользователя
    -  `pwdHash` - хэш пароля, созданный с помощью библиотеки `bcrypt`. Длина - 64 символа.

    Ожидаемое поведение:
    -  На сервере сначала выполняется поиск пользователя с такой почтой в БД. Если такой пользователь существует, то возвращается JSON с сообщением "Пользователь с такой почтой уже существует" (код ошибки - `409 Conflict`)
    -  Если пользователь не найден, то генерируется соль (длина - 16 bit), затем с ее помощью `pwdHash` хэшируется с помощью алгоритма `argon2` (длина - 128 бит, настройки по умолчанию). После этого все данные сохраняются в базу данных. С помощью встроенного в Django `Sessions framework`-а создается сессия (код ответа - `201 Created`).
- `/sign_out`
    `GET` - запрос.

    Параметров нет.
  
    Ожидаемое поведение:
     - С помощью `Sessions framework`-а проверяется, есть ли у данного запроса cookie сессии и является ли она валидной. (**далее данный процесс будет назван "*Валидацией сессии"***)
     - Если сессия не валидна, то возвращается код `401 Unauthorized`.
     - Если сессия валидна, то она инвалидируется, возвращается код `200 OK`. ^fc05e6

### `/api/orders/`
- `/place_order/`
	`POST`-запрос
	
	Параметры:
	- `userId` - ID клиента
	- `cart` - JSON-массив вида
		```
		[
			pizzaId,
			chosenIngredients: {
				"remove": [
					...
				]
				"add": [
					...
				]
			}
		]
		```
	- `location` - адрес доставки
	- `time` - необязательный параметр, ориентировочное время доставки.
	
	Ожидаемое поведение:
	- Выполняется [[#^fc05e6|валидация сессии]]
	- После этого проверяет у каждого элемента JSON-массива pizzaId (существует ли вообще такая пицца), вычисляет общую стоимость с учетом удаленных (???, стоит ли?) и добавленных ингредиентов. Сохраняет результат в БД со статусом не оплачен и не завершён.
	- При успешном создании заказа, возвращает ID-заказа и код `201 Created`.
- `/pay?orderId=?`
	`POST`-запрос
	
	Параметры:
	- `orderId` - ID заказа
	- `paymentMethod` - способ оплаты (СБП / Карта).
	
	Ожидаемое поведение:
	- Выполняется [[#^fc05e6|валидация сессии]]
	- Ищем в БД заказ с таким ID. Если не найден - возвращаем код `404 Not Found`
	- Так как подключение способов оплаты не ожидается, просто обновляем запись заказа в БД, указывая `isPaid=True`
	- При успешной оплате заказа клиенту возвращается код `200 OK`.
- `/cancel_order?orderId=?`
	`GET`-запрос
	
	Параметры:
	- `orderId` - ID заказа
	
	Ожидаемое поведение:
	- Валидация сессии
	- Выставляем `True` в поле `isCancelled` в БД для данного заказа
	- При успешной отмене заказа клиенту возвращается код `200 OK`
